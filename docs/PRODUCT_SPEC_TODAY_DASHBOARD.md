# Product Spec: Today Dashboard (Daily Briefing)

**Feature Owner:** Team 8
**Status:** Implemented
**Last Updated:** 2026-03-05

---

## 1. Overview

The **Today Dashboard** is the home screen of the AI Store Secretary app (`/app`). It serves as a merchant's morning briefing — a single page that summarizes what their AI agent team found, guides them to take action, and tracks progress toward resolving all items.

### Core Principle

> Every finding should guide the merchant to a solution, not just report a problem.

The dashboard doesn't just list issues — it explains what to do, links to where to do it, and confirms when it's done.

## 2. User Stories

| As a... | I want to... | So that... |
|---------|-------------|-----------|
| Merchant | See a personalized morning briefing when I open the app | I immediately know what needs my attention |
| Merchant | Click a top priority and jump to the relevant finding | I can focus on the most impactful issue first |
| Merchant | See a suggested next step with a link to Shopify Admin | I know exactly what to do and where to do it |
| Merchant | Confirm before applying a fix | I don't accidentally change something in my store |
| Merchant | Dismiss insights I've reviewed | I can clear my list and track real progress |
| Merchant | See what my agents handled overnight | I trust the system is working while I sleep |
| Merchant (Advisor mode) | See manual instructions instead of action buttons | I stay in control while still getting AI guidance |

## 4. AI Briefing Card

### Data Source

The briefing is generated by Claude (`app/services/briefing.server.ts`) using all pending findings from enabled agents.

### Briefing Output

| Field | Description |
|-------|-------------|
| `greeting` | Warm one-liner with store name, item count, and total revenue at stake |
| `topPriorities` | Up to 3 items ranked by revenue impact, each with a `findingId` for scroll targeting |
| `autoHandledSummary` | One sentence about what autopilot agents did overnight (or null) |
| `insightHighlight` | One sentence about the most impactful insight (or null) |
| `totalEstimatedRevenue` | Sum of all `estimatedRevenue` values from finding metadata |

### Safeguards

- **FindingId validation**: After AI generates priorities, IDs are validated against actual findings. Invalid/hallucinated IDs are filtered out.
- **In-memory cache**: Briefing is cached for 5 minutes (keyed by finding IDs + statuses) to avoid redundant Claude API calls.
- **Fallback**: If the Claude API fails, a hardcoded briefing is generated from raw finding data (no AI dependency for core functionality).

### Top Priority Click Behavior

1. Merchant clicks a priority item
2. Page smooth-scrolls to the matching finding card
3. Card is highlighted with a blue outline for 3 seconds
4. A context-aware toast appears:
   - Assistant + action_needed: "Review the details, then tap Apply Fix or Dismiss"
   - Advisor mode: "This is an advisory finding — review and decide your next step"
   - Other: "Here's the finding your secretary flagged"

**Technical note**: Click is handled via native `el.onclick` on the `s-box` ref (not React `onClick`) to ensure reliable event handling on Polaris web components.

## 5. Finding Card

The `FindingCard` component renders each individual finding. Its behavior adapts based on the **finding type** and the agent's **trust level**.

### Card Structure

```
+-----------------------------------------------+
| [Agent Badge] [Priority Badge] [Revenue Badge] |
| Title (bold)                                    |
| Description                                    |
|                                                 |
| +-- Suggested next step ----------------------+ |
| | Human-readable guidance text                 | |
| | Open in Shopify Admin →                      | |
| +---------------------------------------------+ |
|                                                 |
| [Apply Fix]  [Dismiss]    ← or advisor banner  |
+-----------------------------------------------+
```

### Priority Badges

| Priority | Label | Tone |
|----------|-------|------|
| 1 | Fix today | critical |
| 2 | Fix today | critical |
| 3 | This week | warning |
| 4 | When you have time | info |
| 5 | When you have time | info |

### Revenue Badge

Shown when `metadata.estimatedRevenue` is present. Displays as `~$X/mo` with `success` tone.

### Suggested Next Step

Shown on **every** card regardless of trust level. Parses the agent's `action` JSON into:

| Action Type | Guidance | Admin Link |
|------------|----------|------------|
| `reorderAlert` | Reorder "{product}" — only {N} days of stock left. | `/products?inventory_quantity_max=20` |
| `applyMetadata` | Update metadata on {N} products to improve AI engine discoverability. | `/products` |
| `addSchema` | Add {schemaType} structured data to your {page}. | `/themes` |
| `expandDescriptions` | Expand descriptions on {N} products to improve conversion and SEO. | `/products` |
| `optimizePage` | Improve the product page for "{handle}" — update hero image and add a FAQ section. | `/products/{handle}` |
| *(plain text)* | Shows the raw action string as guidance | None |
| *(review agent)* | Check the reviews for this product and address recurring complaints. | `/products` |

Admin links use the `shopify://admin` protocol for deep-linking within Shopify Admin.

### Behavior by Trust Level

#### Advisor (read-only)

- Shows "Suggested next step" block with admin link
- Shows info banner: "This agent is in Advisor mode — review the suggestion above and take action manually in Shopify Admin."
- No action buttons

#### Assistant (one-click with confirmation)

- Shows "Suggested next step" block
- Shows **Apply Fix** and **Dismiss** buttons
- **Apply Fix** opens a confirmation box:
  - Repeats the guidance text
  - "Yes, apply" / "Cancel" buttons
  - On confirm: POSTs to `/app/api/agents/findings/{id}/status` with `status: "applied"`
  - Toast: `Done — "{title}" marked as applied`
- **Dismiss**: POSTs with `status: "dismissed"`, toast: "Finding dismissed"

#### Autopilot (auto-executed)

- Findings are auto-applied during agent run (in `run-all` route)
- Cards appear in "Handled Overnight" section as completed items

### Insight Cards ("If You Have Time")

- Show "Suggested next step" if the agent provided an action
- Show a **"Got it"** button to dismiss (previously had no action at all)
- No Apply Fix button (insights are informational)

## 6. Sections

### Needs Your Decision

- Findings with `type: "action_needed"` and `status: "pending"`
- Sorted by priority (1 = highest) then by creation date (newest first)
- Empty state: "All clear — nothing needs your attention right now."

### Handled Overnight

- Findings with `type: "done"` and `status: "pending"`
- Shows what autopilot agents completed automatically
- Empty state: "No automated actions taken yet."

### If You Have Time

- Findings with `type: "insight"` and `status: "pending"`
- Lower-priority observations and opportunities
- Empty state: "No extra opportunities right now."

---

## 7. Check Now

- Primary action button in the page header
- Triggers all enabled agents via `POST /app/api/agents/run-all`
- Shows loading spinner while running
- On complete: revalidates page data, shows toast "All agents finished running"
- Autopilot agents auto-apply their `action_needed` findings during this run

---

## 8. Data Flow

```
[Check Now] → POST /app/api/agents/run-all
                ↓
         Execute all enabled agents
                ↓
         Save findings to DB (upsert by deduplicationKey)
                ↓
         Auto-apply findings for autopilot agents
                ↓
[Page reload] → Loader fetches:
                  - Pending findings (enabled agents only)
                  - Applied findings (for progress count)
                  - Agent settings (trust levels)
                  - Store profile (name for briefing)
                  - Enabled agent IDs
                ↓
         Generate AI briefing via Claude
                ↓
         Validate findingIds
                ↓
         Render dashboard
```

---

## 9. Enabled Agent Filtering

Only findings from **enabled** agents appear on the dashboard. Merchants can enable/disable agents in Settings (`/app/settings`). This prevents disabled agents' stale findings from cluttering the briefing.

## 10. Technical Implementation

### Key Files

| File | Purpose |
|------|---------|
| `app/routes/app._index.tsx` | Today Dashboard page (loader + UI) |
| `app/components/finding-card.tsx` | Finding card with guidance, actions, trust-level behavior |
| `app/components/findings-section.tsx` | Section wrapper that passes trust levels to cards |
| `app/services/briefing.server.ts` | AI briefing generation with caching and fallback |
| `app/services/finding-storage.server.ts` | Finding CRUD (save, query, update status) |
| `app/routes/app.api.agents.findings.$id.status.tsx` | API route for apply/dismiss actions |
| `app/routes/app.api.agents.run-all.tsx` | API route to execute all agents + autopilot logic |

### Dependencies

- **Claude API** (`@anthropic-ai/sdk`): Briefing generation, model `claude-sonnet-4-6`
- **Prisma/MongoDB**: Finding storage, agent settings, store profile
- **Shopify App Bridge**: Toast notifications, embedded app navigation
- **Polaris Web Components**: All UI (`s-page`, `s-box`, `s-badge`, etc.)

## 11. Edge Cases

| Scenario | Behavior |
|----------|----------|
| No findings exist | Greeting says "running smoothly", no priorities, empty sections |
| Claude API is down | Fallback briefing generated from raw data |
| AI returns invalid findingIds | Filtered out before rendering (never shown) |
| All agents disabled | Empty dashboard, agent count shows 0 |
| Finding already dismissed/applied | Not shown (filtered by `status: "pending"`) |
| No action field on finding | "Suggested next step" block is hidden |
| Action is plain text (not JSON) | Shown as-is in guidance block, no admin link |

## 12. Future Considerations

- **Real mutation execution**: Currently "Apply Fix" only marks status as `applied`. Future: execute the actual GraphQL mutation (e.g., update product descriptions).
- **Email briefing**: Infrastructure exists (`briefing.server.ts`, `email-briefing.server.ts`, `app.api.briefing.send.tsx`) but is not yet wired to a scheduler.
- **Undo applied actions**: Allow merchants to revert an applied fix.
- **Finding detail view**: Expandable card or modal with full context, history, and related findings.
